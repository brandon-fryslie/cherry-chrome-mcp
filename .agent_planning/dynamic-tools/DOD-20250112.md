# Dynamic MCP Tool Visibility - Definition of Done (Simplified)
**Date:** 2025-01-12

## Acceptance Criteria

### Two-State Tool Visibility

1. **Not connected state**
   - [ ] When `browserManager.hasConnections()` returns false: only 3 tools visible
   - [ ] Visible tools: `chrome_connect`, `chrome_launch`, `chrome_list_connections`
   - **Test:** Start server fresh, call `tools/list`, verify only 3 tools returned

2. **Connected state**
   - [ ] When `browserManager.hasConnections()` returns true: all 23 tools visible
   - **Test:** Connect to Chrome, call `tools/list`, verify all 23 tools returned

3. **Automatic updates**
   - [ ] After first successful connection: tool list updates without restart
   - [ ] After last connection closed: tool list updates without restart
   - **Test:** Use MCP Inspector, verify tool list changes dynamically

4. **No regression**
   - [ ] All tools function correctly when visible
   - [ ] TypeScript compiles without errors (`npm run build`)
   - **Test:** Run existing test suite, verify no failures

## Out of Scope
- Per-feature visibility (debugger, paused, etc.)
- Tool definition refactoring
- EventEmitter pattern (not needed for 2-state)
- Debouncing
- Description updates
